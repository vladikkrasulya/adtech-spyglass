## 1. Build / Deployment

### 🔴 CRITICAL: `packages/core` Not Bind-Mounted
**Location**: `/srv/DATA/Stacks/adtech-spyglass/docker-compose.yml:13–21`

**Diagnosis**: Only `./public:/app/public` is bind-mounted. The validator core (`packages/core/`) is baked into the image at build time. Every schema/rule change requires `docker compose build` before container restart — no live reload. This blocks rapid iteration on validation logic in development and production hotfixes require full rebuild cycle.

**Impact**: Validator changes demand full rebuild + restart, even for syntax fixes. Slows debugging (you already discovered a typecheck error at `packages/core/categories.js:107` that requires rebuild to verify).

**Fix effort**: S (add volume mount: `- ./packages:/app/packages`)

---

### 🟡 MEDIUM: Manual Cache-Busting via `?v=N` Query Params
**Location**: `/srv/DATA/Stacks/adtech-spyglass/public/index.en.html:50-65` (and `.uk.html`, `.ru.html`)

**Diagnosis**: Asset versioning uses inline `?v=8` bumps on script/style/image links, changed manually. No content-hash or build-time cache-busting. Browser will cache `og-image.png?v=8` forever even after content changes; bumping requires manual hunt across 6 HTML files + testing.

**Impact**: Cache collisions on asset updates. OG image recently regenerated but old `?v=8` may linger in CDN caches or browser caches of users who visited before the bump.

**Fix effort**: M (integrate esbuild/Vite with content-hashed filenames at build time, or use a simple Node script to md5-hash and rewrite links in HTML files during Docker build).

---

### 🟡 MEDIUM: Health Endpoint Does Not Expose Build Metadata
**Location**: `/srv/DATA/Stacks/adtech-spyglass/server.js:134–150`

**Diagnosis**: `/api/health` checks DB connectivity only. ROADMAP Phase 8 requires build SHA + version table state. Current response:
```json
{ "success": true, "db": "ok" }
```
Missing: git commit SHA, build timestamp, validator version, schema version.

**Impact**: Deployments can't easily verify which version is running. Debugging multi-instance rollouts is harder. Load balancers can't signal partial degradation (e.g., one old instance, one new).

**Fix effort**: M (inject `process.env.BUILD_SHA` and `process.env.BUILD_TIME` at build time, expose in `/api/health` response with version table state from `db.js`).

## 3. Architecture

### 🟡 MEDIUM: Server & Client Validation Boundary Not Isolated
**Location**: `/srv/DATA/Stacks/adtech-spyglass/packages/core/` (shared), `server.js:611` (server-side), `public/spyglass.app.js:1000+` (client-side, planned).

**Diagnosis**: The validator core lives in `packages/core/` and is required by both `server.js` (POST `/api/analyze`) and will be used in the browser (future: client-side validation). The abstraction works — core has no Node-only deps — but there's no clear versioning or changelog for breaking changes to the core API.

**Example**: If you change the `validate(payload, opts)` signature in `packages/core/index.js`, both the server and future browser code must update in sync. No versioning boundary means a bad push breaks both.

**Impact**: Risk of skew between server and client validation if the core API is refactored without a clear deprecation strategy. Not an issue *right now* but will bite before Phase 4 (open-source publish).

**Fix effort**: M (add a CHANGELOG to `packages/core/`, document the public API surface clearly with JSDoc, consider semver tagging once the package is published to npm).

### 🟡 MEDIUM: 3-Locale HTML File Architecture Drifts from Client i18n
**Location**: `public/index.{en,uk,ru}.html` (static files) vs `public/i18n.js` (client string registry) vs `packages/core/messages/{uk,en}.json` (validator messages).

**Diagnosis**: Three i18n systems coexist:
1. **Static HTML per locale** (SEO strategy): Each `.html` file is hardcoded for one language. UI copy (button labels, titles) is baked in per file.
2. **Client-side i18n.js** (~140 keys): JavaScript strings resolved at runtime for dynamic UI (modals, toasts, tooltips).
3. **Validator messages** (`packages/core/messages/`): Validator findings are localized via JSON files.

The systems are *loosely coupled* — if you add a new UI string, you must update:
- All 3 HTML files (6 total: index + about)
- i18n.js (one place)
- Validator messages if it's a finding label (one place)

ROADMAP Phase 3 aimed to consolidate this but the old static-file approach remains.

**Impact**: Drift happens. A feature lands with UK + EN copy in HTML but RU is missing until someone notices. "Check all 6 files" is manual and error-prone. Validator messages sometimes use a different tone than UI.

**Fix effort**: L (Phase 3 effort — extract all hardcoded strings from HTML into a shared i18n system, generate the 3 HTML files from a template + locale data at build time; or use client-side rendering for everything and keep the static files as fallbacks for NoJS).
