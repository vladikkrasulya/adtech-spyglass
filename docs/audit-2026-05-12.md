## ✅ CRITICAL — already fixed today (v0.41.0 → v0.41.2)

### C1. `Cache-Control: no-cache` on every static asset
`server.js:441` (pre-fix). Every JS/CSS/font/image response carried
`no-cache`, forcing the browser to revalidate every asset on every page
load. With content-hash `?v=…` query strings already invalidating
caches on actual content change, this was double work — 25+ assets ×
~300 ms round-trip via CF Tunnel = ~7-10 s pure handshake on every
visit. **Root cause of the "20 seconds to load" user report.**

**Fix:** branch by URL — `public, max-age=31536000, immutable` for
hash-versioned URLs (the hash itself is the cache key); `no-cache`
preserved for HTML entry points and unversioned legacy assets.
Verified via `curl -I`. Server.js:441-455.

### C2. `backdrop-filter: blur(12px)` on sticky topnav
`/srv/DATA/Stacks/kyivtech-portal/public/design-system.css:503-511`.
Sticky positioning + 12px Gaussian blur = browser composites + blurs
the entire content area sliding under the topnav on every scroll
frame. GPU cost ~O(radius²). On 120 Hz monitor with ~8.3 ms frame
budget, this alone exceeded the budget → dropped frames → perceived
"~20 Hz" jank on a *static* page.

**Fix:** Spyglass-specific override (no portal impact) reduced blur to
4px (~9× cheaper) via inline `<style>` in account.\* + about.\*
HTML files. `transition: all 0.15s` on `.cab-radio` replaced with
explicit `background-color, border-color, color` to avoid the
"transition: all" property-monitoring overhead.

## 🔴 HIGH — open

### H1. Synchronous `fs.readFileSync` + recursive SHA1 in `fileHash()`
**File:** `server.js:313-352` (`fileHash`), called from
`rewriteAssetVersions` (`server.js:291-310`).

```js
if (ext === '.js' || ext === '.html') {
  // Not mtime-cached because a child mtime change must invalidate this
  // entry; tracking that explicitly is more complex than just
  // recomputing on each request.
  const buf = fs.readFileSync(filepath);
  const rewritten = rewriteAssetVersions(buf.toString('utf8'),
    ext === '.html' ? 'html' : 'js', v);
  return crypto.createHash('sha1').update(rewritten).digest('hex')
    .slice(0, 8);
}
```

Every request to a .js or .html file walks the import tree, reads each
.js / .html descendant **synchronously**, computes SHA1. For `/uk/`
this means 17+ sync FS reads on the main-thread event loop per
request. Locally ~1 ms total; under concurrent load or cold cache it
serialises the event loop and lengthens TTFB.

**DS notes a TOCTOU race in passing:** the `.css`/image branch does
`fs.statSync` then later `fs.readFileSync` — non-atomic. If the file
changes between, the cached entry holds the *old* hash for the *new*
content until next mtime change. Low real-world risk because we
content-hash terminal assets and the mtime moves with content; but
worth knowing.

**Fix sketch (per DS):** add an mtime cache for .js / .html with
reverse dependency-graph invalidation:

```js
const _jsHtmlCache = new Map();      // path → {mtimeMs, hash, deps}
const _dependencyGraph = new Map();  // child → Set(parents)

// On rewrite: record `deps` from parsed imports.
// On `fs.watch(child) → 'change'`: invalidate every parent in the graph.
```

This collapses the per-request cost from O(n) sync I/O to O(1) Map
lookup, with full correctness as long as `fs.watch` is reliable
(chokidar fallback recommended for cross-platform).

### H2. 17 blocking `<script src>` tags without `defer`
**File:** `public/index.uk.html:298-334` (en/ru analogous).

```html
<script src="/version.js?v=…"></script>
<script src="/spyglass-crypto.js?v=…"></script>
<script src="/i18n.js?v=…"></script>
<script src="/lang-switch.js?v=…"></script>
<script type="module">
  import inspectorModule from '/modules/inspector/index.js';
  …
</script>
<!-- 12 more classic scripts -->
```

Classic `<script src>` blocks the HTML parser until the script is
downloaded and executed. With 12+ classic scripts after the inspector
module, parser is forced to serialise. The inspector module is `type=
module` (deferred by spec), but the 12 following classic scripts
serialise behind it.

**DS verdict on adding `defer` to all 12:** safe. They already wait on
the `kt:inspector-ready` custom event (dispatched after the inspector
module's async mount). `defer` runs scripts after DOMContentLoaded in
document order — the event listener attachment is unaffected.

**Risk:** order-sensitive shared globals. If `i18n.js` exposes
`window.t()` that any of the 12 classic scripts touches at top-level
(not inside an event handler), defer's reordering must preserve order
— which `defer` does, so this is fine. **Recommend manual smoke-test
after defer is added** (open share modal, embed dialog, shortcuts
help — each was a classic script).

### H3. `JSON.parse(userInput)` without try/catch in 6 client modules
**Files (no try/catch wrapping `JSON.parse`):**
- `public/modules/mirror/index.js:108-109` — `bidReq`/`bidRes` from
  textarea
- `public/modules/simulate/index.js:38` — same
- `public/modules/save-sample/index.js:95` — same
- `public/modules/live/index.js:153` — `ev.data` from SSE
  (low risk — server-controlled)
- `public/modules/stream/index.js:133` — same SSE
- `public/modules/dialects/index.js:259` — file upload import

User pastes invalid JSON → `SyntaxError` thrown → bubbles up → click
handler silently fails or visible exception breaks the modal. UX
feels broken; root cause is one bad character.

**DS upgrades this from MEDIUM → HIGH** because it surfaces directly
to end users with no recovery affordance.

**Fix:** wrap each call:
```js
let parsed;
try { parsed = JSON.parse(value); }
catch (_) {
  toast(t('toast.invalid_json'), 'error');
  return;
}
```

### H4. `spyglass.app.js = 192 KB / 4510 lines` monolith
**File:** `public/spyglass.app.js`.

Single bundle loaded by the inspector module. No code-splitting per
feature (save-sample modal, share modal, embed modal, behavior
analysis, intel banner, recovery flow, password reset, etc. — each
~5-20 KB but all in one file). Parse + compile time on slower
hardware is non-trivial.

**Not urgent now** that browser cache holds it long-term (post-C1
fix). But every feature touch invalidates the whole bundle.
**Recommendation:** plan a code-split sprint where each modal /
feature becomes its own ES module loaded on first use. Matches the
existing `/modules/<name>/index.js` pattern that newer features
(intel, recovery, share, embed) already follow.

## 🟡 MEDIUM — open

### M1. `injectModuleBundleHashes()` does sync directory reads
**File:** `server.js:489-525` (approx).

For every `__<MODULE>_BUNDLE_HASH__` token in a .js file response, the
server does a directory listing + sync read of every file in
`public/modules/<module>/` to compute the bundle hash. Less frequent
than H1 (only when a module file is requested AND contains a bundle-
hash token) but the same sync-I/O smell.

**Same mtime-cache fix as H1 applies.** Watch the module directory,
invalidate the hash on any change.

### M2. Duplicated inline `<style>` blocks in account.{uk,en,ru}.html
~440 lines of CSS duplicated 3× (one per locale variant). Maintenance
burden + 3× the bytes shipped. The actual styling doesn't differ by
locale.

**Fix:** extract to a separate `account.css`, load via `<link
rel="stylesheet" href="/account.css?v=…">` (cache-friendly), keep only
locale-specific overrides inline if any. Same applies to
about.{uk,en,ru}.html and any other 3-way duplication.

### M3. `inspector.css = 62 KB / 2500 lines`
Loaded async via dynamic `<link>` by the inspector module — that
prevents render-blocking, but parse cost is still ~10-20 ms on
mid-range hardware. Acceptable for now, but ripe for splitting if
inspector grows further.

### M4. Potential WebSocket / SSE message-queue unboundedness
**Not deeply audited.** `modules/stream/handler.js` has the right
cleanup hooks (heartbeat + listener removed on close, idempotent
`cleaned` boolean). `modules/live/handler.js` should be reviewed for
similar discipline. **DS flagged this category as "unreviewed";
recommend a focused look next time.**

### M5. `filter: blur(22px)` on `.preview-safe iframe`
**File:** `public/modules/inspector/inspector.css:558`.

Only active when a user has loaded a creative AND not clicked "Show
creative". GPU cost is large but path is rare. DS downgraded to LOW.
Consider dropping to `blur(12px)` if jank shows up specifically when
previewing — still hides content adequately.

## ⚪ Categories that came back CLEAN (no findings)

- **SQL injection:** All queries use `db.prepare(?)` with parameter
  binding. `db.exec(...)` is only used for static schema migrations.
- **Path traversal in static serving:** `server.js:409-417` normalises
  + strips `..` segments + verifies `path.resolve` is inside
  `PUBLIC_DIR`. Looks correct.
- **Crypto correctness:** AES-GCM, bcrypt, PBKDF2 patterns in
  spyglass-crypto.js and auth.js use proper IV/salt generation and
  comparison. (Not deep-reviewed — see "Not audited" below.)
- **Process-level error handling:** `uncaughtException` +
  `unhandledRejection` + SIGINT/SIGTERM shutdown all wired.
- **N+1 queries:** None found. List endpoints use single-query
  `SELECT … COUNT(*)` subselects or `JOIN` on indexed columns.
- **Empty `catch {}` blocks** that silently swallow errors: none in
  server-side code.
- **`Math.random()` in security contexts:** none — all crypto-relevant
  IDs use `crypto.randomBytes` or `crypto.randomUUID`.
- **Memory growth in long-lived caches:** rate-limiter buckets,
  WeakMap user-dialect cache — all bounded or auto-collected.
- **Foreign keys + WAL + busy_timeout:** all set on DB init (db.js:31-36).
