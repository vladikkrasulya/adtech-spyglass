Spyglass — how to use it

A private inspector for OpenRTB traffic. Paste JSON, get a breakdown: whether the request/response is well-formed, which fields are required, whether request↔response crosscheck holds, which oRTB version you're looking at, and what the IAB spec says about a specific clause.

Self-hosted on bare metal in Kyiv. No ads, no trackers, no third-party data sharing.

What's new · 2026-05-10

Eleven releases shipped in one day (v0.25 → v0.35). Quick tour of the new surfaces — full detail in the sections below or in CHANGELOG.md.

🪞 Mirror — canonical pair, one click

Hit mirror ↔ in the header (or hotkey M) and Spyglass generates the canonical counterpart of your paste. BidRequest in → minimal-valid BidResponse out. BidResponse in → BidRequest the response would fit. Output is self-tested through the existing validate() + crosscheck() loop, so it's always clean. Two modes: minimal (required fields only) and best-practice (with recommended: schain, dchain, DSA, GDPR consent, device.sua).

With both panes filled, the modal also shows a diff: your version vs the canonical one, colour-coded (≠ different, + canonical-only, − you-only). 🔗 share button generates a permalink carrying the pair — handy for SSP calls.

📡 Live RTB stream

📡 live button opens a modal subscribed to a synthetic RTB sample stream. Each row: timestamp, kind (REQ/RES), source filename, optional banner size. Pause/resume button. Click a row to load that sample into the matching editor and close the modal. Foundation for the upcoming Stream platform.

👁 Finding details

Every validator finding now has a ▾ expander. Click reveals: JSON path, your current value at that path (extracted from the paste), severity in plain language ("error → exchange rejects"; "warning → reduces fill-rate"; "info → best-practice note"), spec link, and canonical rule id.

🛡 Behavior corpus + Confusion matrix

On the behavior tab, 💾 save as corpus appears when there are probe events and you're signed in. Save the event-stream with a label (legitimate / fraud / ambiguous) — your cabinet has a section for all entries. Foundation for Chapter B (behaviour calibration).

Off the corpus, the confusion matrix auto-computes precision, recall, F1 for each of the 12 detectors against your own labelled data. Spot low-precision patterns (red row) that need tuning, or stable ones (≥90% green).

🤖 Bid simulator (local LLM 3-strategy)

🤖 simulate button. Local qwen2.5:3b runs three DSP strategies: aggressive (max scale, +30-50% over floor), conservative (ROAS guard, +5-15%), quality (premium only, +50-80% or skip). Privacy: the LLM sees only an 8-field metadata summary (formats, sizes, geo, currency, floor, surface, device-type, auction-type) — bid VALUES never reach the LLM.

📦 Specimen replay endpoint

POST /api/v1/replay accepts an array of samples, runs validate + crosscheck + behavior on each, returns per-sample results plus an aggregate (statusCounts, topFindings, totalFindings). One call instead of N round-trips. Capped at 100 samples per call. Foundation for CI fixture grading, archive replay, partner audit pipelines.

👤 Cabinet — new 7-section layout

Cabinet (/account) restructured: sticky sidebar with 7 sections — Profile · Library · Activity · Behavior corpus · Preferences · Security · Danger zone. Scroll-spy highlights the active section as you scroll. URL hash updates for deep links: /account#corpus jumps straight in. Mobile (≤880px) collapses the sidebar to a horizontal tab-chip bar.

English and Russian about pages shipped in v0.32 (previously stubs). What Spyglass does

BidRequest / BidResponse breakdown

  • Structure validation — required fields, types, formats (imp[] non-empty, id string, tmax number, cur[] ISO-4217)
  • Ad-type support — banner, video, native (1.1 + 1.2), audio. VAST protocols in video.protocols, app vs site, rewarded (imp.rwdd)
  • Crosscheck request↔response — bid.impid → imp.id match, creative format aligns with imp.banner/video, price ≥ bidfloor, creative size matches imp.banner.format[]
  • Native crosscheck — matches request asset-ids against response asset-ids, catches missing/extra assets, invalid JSON in adm

Detect & inform

  • Auto-detect version — analyses signature fields (imp.rwdd, device.sua, regs.gpp_sid, source.schain) and decides whether the payload is oRTB 2.5 / 2.6 baseline / 2.6-202211 / 2.6-202309 / 3.0. Returns confidence and the list of detected signals.
  • Spec deep-links — every finding carries a link to the exact paragraph in the IAB spec (§3.2.10, §5.4)
  • Severity levels — error / warning / info. UI highlights with an icon.

Vendor overlays

  • IAB (default) — clean IAB spec, no extensions.
  • Vendor overlays — optional rule overlays for specific SSPs/DSPs (their ext.*, custom macros, vendor-specific constraints). Activated via ?dialect=<vendor> query parameter. The public UI stays generic.

Ad preview

If bid.adm contains an HTML creative — Spyglass renders it in a sandboxed iframe (sandbox="allow-scripts", no allow-same-origin). You see how it looks, with no risk to your session.

Behavior tab — in-creative fraud detection

Spyglass renders every creative in a sandboxed iframe with an instrumentation probe inside. The probe catches 11 patterns of suspicious behaviour — from invisible click traps to main-thread freezes — and the Behavior tab next to Validation / Crosscheck shows structured findings with severity and destination URL. Architecture deep-dive in the Behavior Analysis & Anti-Fraud section below.

Sharing & ergonomics

  • Share link — copies a self-contained URL like spyglass.kyivtech.com.ua/?#req=…&res=…. The payload is deflate-compressed + base64url-encoded into the URL hash fragment, which never reaches the server (zero-knowledge by URL design). Opening the link auto-populates both panes and re-runs the analysis.
  • Download bundle — packages request + response + findings + crosscheck into a single JSON file (spyglass-{date}-{6-hex}.json) for attaching to a bug ticket or pasting into chat without juggling two large blobs.
  • Keyboard shortcuts — ? shows the cheat sheet, Ctrl+Enter runs analysis, Ctrl+S saves to the library (auth-gated), Esc closes any modal.

Personal workspace (with an account)

  • Saved samples — real requests/responses with notes, grouped by partner
  • Partners — catalogue of SSPs/DSPs/AdNetworks you work with
  • Zero-knowledge encryption — everything you save is encrypted in your browser with a key derived from your password. The server cannot decrypt your samples even if it wanted to. There's a recovery key for the lost-password case — without it, access is gone forever.

Spyglass Intelligence — local AI assistant

If the Spyglass admin has wired up a local Ollama instance with the qwen2.5:3b model, you get an AI helper that names new field clusters in Discovery mode and explains the purpose of individual ext.* fields. Everything runs on the same machine as Spyglass, never the public internet:

  • 100% local — the model lives in a Docker container next to Spyglass. No token ever leaves the LAN.
  • Field paths only — bid values never enter the prompt. Only field names and char-class hints (in addition to the PII filter).
  • On-demand — the AI does not analyse anything automatically. Click the 🤖 Suggest button in the Dialect Builder and you get an answer. Don't click — nothing happens.
  • Knowledge Base as context — the model receives 1-2 reference field examples from our embedded format library (push, banner, native…) so cluster names land grounded in real-AdTech vocabulary, not priors.
  • Fail-open — if Ollama is offline or breaks, the AI buttons hide quietly. The rest of Spyglass works as before. Without AI you only lose auto-naming of clusters; everything else continues.

Safe Demo Mode — blur for screenshots

Append ?demo=safe to the URL and Spyglass:

  • Applies CSS blur to the creative preview pane (silhouette visible, content not)
  • Masks domain names in the summary panel (***rt.com instead of the real domain)
  • Doesn't touch validation — the engine works at 100%; only the visual layer hides

Useful for documentation screenshots, client decks, or when you need to show the tool without exposing specific advertising partners — particularly relevant for adult / casino / gambling traffic where even a logo silhouette is a leak.

How the analysis works — three layers

Spyglass looks at your payload from three angles at once. Each layer answers its own question; together they give the full picture.

Layer 1 — Static (schema)

"Is this JSON well-formed?" We parse the payload, validate against the IAB spec (oRTB 2.5 / 2.6 / 3.0, Native 1.x, VAST protocols), check required fields, types, formats. Output is a list of findings with severity (error / warning / info), a JSON path to the offending location, and a deep-link to the relevant IAB doc paragraph. This is the fastest layer — milliseconds on pure-JS rules, no side-effecting calls.

Layer 2 — Behavior (sandbox iframe)

"What does this creative actually do in the browser?" We render bid.adm in a sandboxed iframe (sandbox="allow-scripts", without allow-same-origin) with an instrumentation probe inside. The probe catches suspicious behaviour: invisible click-traps, frame-bust attempts, frozen-thread heartbeats, permission abuse (Notification, geolocation, getUserMedia), miner signatures in code, click-skim probes (fake click events that anticipate the user's real click). The engine produces structured findings — same shape as Static, plus severity and destination URL. Details in the Behavior Analysis & Anti-Fraud section.

Layer 3 — Knowledge Base (embedded encyclopedia)

"Have I seen this format before?" Inside packages/core/knowledge_base/ we keep a curated collection of canonical oRTB / JsonFeed payloads — banner / video / audio / native / push / pops / inpage / inapp / dooh / ctv. The format-detect engine tags your payload against this base (a VAST banner inside an Android InApp slot is a typical example), and those tags serve two purposes at once: they render as chips in the UI, and they get injected into the local LLM's prompt as few-shot context, so the AI answers in real-market vocabulary rather than priors. The base is license-clean (synthesised samples + Apache-2.0 / MIT material from Prebid.js and IAB), without any real bid-stream content.

Behavior Analysis & Anti-Fraud

JSON validation answers the question "is the request/response well-formed?". The Behavior engine answers a fundamentally different one: "what does this creative actually do in the user's browser?". It's the second layer — runtime analysis of the rendered HTML/JS that you can't see from payload inspection alone.

How it works

Spyglass renders the creative in a sandboxed iframe with sandbox="allow-scripts" — no allow-same-origin, no allow-top-navigation, no allow-popups. That's enough to block most malicious actions. But blocking ≠ visibility: a creative trying to fraud against a locked-down sandbox simply stays silent — the user never sees that anything was wrong.

So before the creative HTML parses, a creative-probe runs in the iframe (~830 lines of JavaScript, 15 instrumentation hooks). The probe wraps navigation APIs (window.open, Location.assign / replace / href=), programmatic clicks (HTMLElement.click, HTMLFormElement.submit), event listeners, plus a PerformanceObserver(longtask), a PerformanceObserver(resource), and a 1Hz heartbeat. Every suspicious action — postMessage to the parent at the moment of detection.

The probe also measures real element geometry via getBoundingClientRect() + getComputedStyle() at click time — that's how it detects invisible overlay traps (opacity < 0.05 on an element covering > 50% of viewport), close buttons that are too small (< 44×44 IAB minimum), and pixel-perfect synthetic clicks from pixelbots.

The parent window runs these events through the engine in packages/core/behavior/ — a pure-data function that matches patterns and emits structured findings with severity (error / warning / info), destination URL, and metadata (gesture lineage, threshold breached, etc.). Engine is browser- and-Node compatible: it runs in the browser via /api/analyze-behavior and on the server for archive replay.

Four detection vectors

1. Misclick traps

The creative mounts an invisible element (opacity=0, transparent background) covering the full viewport above visible content. Any user click lands NOT where they intended — in the trap, which logs the click as a conversion or redirects to a landing page. The probe measures the click target's bounding rect and computed style: if the rect covers > 50% of viewport AND opacity < 0.05 → behavior.trap.invisible_overlay (severity ERROR).

2. Bot patterns

Pixelbot scripts read button centers via getBoundingClientRect() and dispatch click events at that exact coordinate — but event.isTrusted on a synthesized click is always false (setting it to true requires WebDriver / CDP, which a sandboxed creative has no access to). The probe catches three patterns:

  • center-synth — distance < 0.5px from center + isTrusted=false (behavior.bot.center_synth, ERROR)
  • click-burst — 3+ clicks within a 200ms sliding window (behavior.bot.click_burst, ERROR — humans don't tap that fast)
  • phantom-click — a click with zero preceding mousemove or touchstart since the creative loaded (behavior.bot.phantom_click, WARNING — typical of headless browsers)

3. Malicious ads — frame busting & auto-redirects

The trickiest vector: the creative tries to escape the sandboxed iframe up to the top frame. Via <a target="_top"> anchors, <form target="_top"> submission, or setTimeout-driven auto-redirect. The sandbox blocks the actual navigation (no allow-top-navigation), but the probe sees the attempt. The engine tracks user-gesture lineage via navigator.userActivation + a manual _lastGestureAt snapshot and splits into four sub-patterns:

  • frame_bust_anchor — click on <a target="_top">; ERROR without gesture lineage, WARNING within the 500ms gesture-grace
  • frame_bust_form — form submit with target="_top"; always ERROR (forms with top-level navigation have no legitimate use in a banner)
  • auto_redirect — navigation with an empty event-stack and no recent gesture (ERROR)
  • late_redirect — gesture happened, but navigation fired setTimeout > 500ms later (WARNING — classic cloaking: user clicked something visible, then setTimeout ran the real navigation outside the gesture lineage)

4. Heavy ads

Creatives that block the main thread excessively or pull megabytes of resources. The probe uses PerformanceObserver(longtask) to accumulate CPU blocking time and PerformanceObserver(resource) for transferSize. Thresholds mirror Chrome Heavy Ad Intervention:

  • heavy_ad_cpu — > 60s cumulative blocking OR > 4s in any 30s sliding window (ERROR)
  • heavy_ad_network — > 4 MB cumulative sub-resources (ERROR; IAB display-ad weight cap)
  • frozen_thread — separate mechanism via the parent watchdog: the probe sends a 1Hz heartbeat, the parent monitors lag — if > 3.5s without a signal, a synthetic event is injected (a frozen probe can't report its own freeze — an infinite loop blocks both setInterval and postMessage, so detection has to live outside the iframe). ERROR.

All of this runs client-side, zero backend load. The engine in packages/core/behavior/ is pure-data; findings render in the Behavior tab alongside Validation / Crosscheck. The sandbox blocks the fraud action itself; we make the attempt visible — so you can attach it to a bug ticket and go back to the creative supplier with a screenshot.

What Spyglass doesn't do

Limitation Why
Doesn't make real bids It's a JSON inspector, not an RTB client. No exchange connections.
Doesn't auto-fix JSON We point at errors + link to the spec paragraph, but don't edit your payload. Fixing the JSON is your job.
Doesn't analyse video / image pixels The sandboxed iframe catches HTML/JS behaviour (clicks, redirects, permission requests, malware patterns). What the picture or video frame actually shows — out of scope.
AI doesn't guess unknown-field logic without your command Suggest only fires when you click 🤖 in the Dialect Builder. No background analysis, no "AI knows best" — you control every call.
Doesn't expand macros (${AUCTION_PRICE}) at runtime We only check statically — whether a macro is valid, whether it's spelled correctly.
Doesn't render real third-party creatives The iframe preview shows the local adm only — no VAST player, no external-tag fetching.
Doesn't enforce full oRTB 2.6 strictness Current baseline is 2.5 plus 2.6 signal detection. Strict 2.6 validation — Phase 2.
Doesn't save samples for anonymous users Anonymous = lives in the browser tab only; reload = empty.
Doesn't log IP/UA on /api/analyze Without auth — minimal trace (CF Tunnel access-log, not our backend).
Doesn't optimise bid strategy / eCPM / yield It's an inspector, not a yield manager.
Doesn't parse TCF/GPP consent strings Planned as a separate tab in kyivtech.com.ua/tools/.
Doesn't validate ads.txt / sellers.json Separate tool, not mixed into the oRTB inspector.

What data Spyglass uses for analysis

Locally (in-process, on our server)

  1. Your JSON — everything you paste into BID REQUEST / BID RESPONSE. Parsed, validated.
  2. IAB OpenRTB 2.5 / 2.6 / 3.0 specs — bundled markdown, used for generating specRef links
  3. Native 1.1 / 1.2 specs — bundled, used for asset validation
  4. VAST 4.x protocol metadata — used for video.protocols validation
  5. spec-refs.json — finding-id → IAB spec anchor map
  6. messages/{uk,en,ru}.json — localised finding messages
  7. Vendor dialect overlays (only with ?dialect=<vendor>) — bundled, no calls to vendor APIs

NOT used

  • ❌ Zero third-party API requests during analysis
  • ❌ No browser fingerprinting / tracking pixels
  • ❌ No cookies for anonymous users (only a session cookie after login)
  • ❌ No external CDNs — everything self-hosted
  • ❌ No analytics (Google, Mixpanel, etc.)

Analysis flow

[your JSON]
    ↓
parseJSON → validate(structure) → detectVersion → crosscheck
    ↓
findings[]: { id, level, path, params, msg, specRef }
    ↓
server formats localised text and returns 200 JSON

Everything happens synchronously in a single Node.js process, with no DB access, no external fetch, nothing leaving the box.

What's stored on the server

Anonymous visitors

Nothing. /api/analyze doesn't write to the DB, doesn't log payloads, doesn't store call history. Cloudflare Tunnel in front of us records the request itself (IP, path, status) — that's a standard access-log, not our backend.

Registered users

What How
email + password hash bcrypt, 12 rounds
Session in-memory (TTL 30 days). Server restart → everyone signs out
Saved samples (bid_req, bid_res) AES-GCM ciphertext. The server sees bytes, but cannot decrypt.
KDF salt + wrapped DEK + IVs Yes. Useless to an attacker without your password.
Notes / sample metadata Same encryption
Partners (label) Same encryption
email_verified_at timestamp (only after verify-email)

Implementation: password on the client → PBKDF2 600k SHA-256 → KEK → unwraps the stored DEK → DEK encrypts content. The server never sees your password and cannot derive the DEK. When you change your password the DEK gets re-wrapped (not regenerated), so old samples stay accessible.

How to read the result

Each finding looks like this:

{
  "id": "imp.banner.size_required",
  "level": "error",
  "path": "imp[0].banner",
  "params": { "num": 1 },
  "msg": "imp[1] banner: w+h or format[] required",
  "specRef": "https://iabtechlab.com/...#322"
}
  • level — error (required-rule violation) / warning (suspicious, not forbidden) / info (note)
  • path — JSON-path to the offending field in your payload
  • msg — human-readable text, localised
  • specRef — direct link to the relevant IAB-spec paragraph

Overall status:

  • errors — at least one level: error
  • warnings — only warnings/info, but not clean
  • clean — nothing found

Quick example

Paste into the BID REQUEST field:

{
  "id": "test-001",
  "imp": [
    { "id": "1", "banner": { "w": 300, "h": 250 }, "bidfloor": 0.5 }
  ],
  "site": { "id": "publisher-abc", "domain": "example.com" },
  "device": { "ua": "Mozilla/5.0", "ip": "1.2.3.4" }
}

Hit analyze stream (or Ctrl+Enter) — you'll see detected version 2.5, a few warnings (no at, tmax, cur), spec-refs to IAB §3.2.1, §3.2.4.

Add into BID RESPONSE:

{
  "id": "test-001",
  "seatbid": [
    { "bid": [
      { "id": "b1", "impid": "1", "price": 1.5,
        "adm": "<html>creative</html>",
        "adomain": ["example.com"], "w": 300, "h": 250 }
    ]}
  ],
  "cur": "USD"
}

Now crosscheck is active — you'll see impid → imp.id match, size match, price > bidfloor check, and the ad preview rendering in the bottom-right card (sandboxed).
