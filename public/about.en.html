Spyglass — how to use it

A private inspector for OpenRTB traffic. Paste JSON, get a breakdown: whether the request/response is well-formed, which fields are required, whether request↔response crosscheck holds, which oRTB version you're looking at, and what the IAB spec says about a specific clause.

Self-hosted on bare metal in Kyiv. No ads, no trackers, no third-party data sharing.

What Spyglass does

BidRequest / BidResponse breakdown

  • Structure validation — required fields, types, formats (imp[] non-empty, id string, tmax number, cur[] ISO-4217)
  • Ad-type support — banner, video, native (1.1 + 1.2), audio. VAST protocols in video.protocols, app vs site, rewarded (imp.rwdd)
  • Crosscheck request↔response — bid.impid → imp.id match, creative format aligns with imp.banner/video, price ≥ bidfloor, creative size matches imp.banner.format[]
  • Native crosscheck — matches request asset-ids against response asset-ids, catches missing/extra assets, invalid JSON in adm

Detect & inform

  • Auto-detect version — analyses signature fields (imp.rwdd, device.sua, regs.gpp_sid, source.schain) and decides whether the payload is oRTB 2.5 / 2.6 baseline / 2.6-202211 / 2.6-202309 / 3.0. Returns confidence and the list of detected signals.
  • Spec deep-links — every finding carries a link to the exact paragraph in the IAB spec (§3.2.10, §5.4)
  • Severity levels — error / warning / info. UI highlights with an icon.

Vendor overlays

  • IAB (default) — clean IAB spec, no extensions.
  • Vendor overlays — optional rule overlays for specific SSPs/DSPs (their ext.*, custom macros, vendor-specific constraints). Activated via ?dialect=<vendor> query parameter. The public UI stays generic.

Ad preview

If bid.adm contains an HTML creative — Spyglass renders it in a sandboxed iframe (sandbox="allow-scripts", no allow-same-origin). You see how it looks, with no risk to your session.

Personal workspace (with an account)

  • Saved samples — real requests/responses with notes, grouped by partner
  • Partners — catalogue of SSPs/DSPs/AdNetworks you work with
  • Zero-knowledge encryption — everything you save is encrypted in your browser with a key derived from your password. The server cannot decrypt your samples even if it wanted to. There's a recovery key for the lost-password case — without it, access is gone forever.

What Spyglass doesn't do

Doesn't make real bidsIt's a JSON inspector, not an RTB client. No exchange connections.
Doesn't expand macros (${AUCTION_PRICE}) at runtimeWe only check statically — whether a macro is valid, whether it's spelled correctly.
Doesn't render real third-party creativesThe iframe preview shows the local adm only — no VAST player, no external-tag fetching.
Doesn't enforce full oRTB 2.6 strictnessCurrent baseline is 2.5 plus 2.6 signal detection. Strict 2.6 validation — Phase 2.
Doesn't save samples for anonymous usersAnonymous = lives in the browser tab only; reload = empty.
Doesn't log IP/UA on /api/analyzeWithout auth — minimal trace (CF Tunnel access-log, not our backend).
Doesn't optimise bid strategy / eCPM / yieldIt's an inspector, not a yield manager.
Doesn't parse TCF/GPP consent stringsPlanned as a separate tab in kyivtech.com.ua/tools/.
Doesn't validate ads.txt / sellers.jsonSeparate tool, not mixed into the oRTB inspector.

What data Spyglass uses for analysis

Locally (in-process, on our server)

  1. Your JSON — everything you paste into BID REQUEST / BID RESPONSE. Parsed, validated.
  2. IAB OpenRTB 2.5 / 2.6 / 3.0 specs — bundled markdown, used for generating specRef links
  3. Native 1.1 / 1.2 specs — bundled, used for asset validation
  4. VAST 4.x protocol metadata — used for video.protocols validation
  5. spec-refs.json — finding-id → IAB spec anchor map
  6. messages/{uk,en}.json — localised finding messages
  7. Vendor dialect overlays (only with ?dialect=<vendor>) — bundled, no calls to vendor APIs

NOT used

  • ❌ Zero third-party API requests during analysis
  • ❌ No browser fingerprinting / tracking pixels
  • ❌ No cookies for anonymous users (only a session cookie after login)
  • ❌ No external CDNs — everything self-hosted
  • ❌ No analytics (Google, Mixpanel, etc.)

Analysis flow

[your JSON]
    ↓
parseJSON → validate(structure) → detectVersion → crosscheck
    ↓
findings[]: { id, level, path, params, msg, specRef }
    ↓
server formats localised text and returns 200 JSON

Everything happens synchronously in a single Node.js process, with no DB access, no external fetch, nothing leaving the box.

What's stored on the server

Anonymous visitors

Nothing. /api/analyze doesn't write to the DB, doesn't log payloads, doesn't store call history. Cloudflare Tunnel in front of us records the request itself (IP, path, status) — that's a standard access-log, not our backend.

Registered users

WhatHow
email + password hashbcrypt, 12 rounds
Sessionin-memory (TTL 30 days). Server restart → everyone signs out
Saved samples (bid_req, bid_res)AES-GCM ciphertext. The server sees bytes, but cannot decrypt.
KDF salt + wrapped DEK + IVsYes. Useless to an attacker without your password.
Notes / sample metadataSame encryption
Partners (label)Same encryption
email_verified_attimestamp (only after verify-email)

Implementation: password on the client → PBKDF2 600k SHA-256 → KEK → unwraps the stored DEK → DEK encrypts content. The server never sees your password and cannot derive the DEK. When you change your password the DEK gets re-wrapped (not regenerated), so old samples stay accessible.

How to read the result

Each finding looks like this:

{
  "id": "imp.banner.size_required",
  "level": "error",
  "path": "imp[0].banner",
  "params": { "num": 1 },
  "msg": "imp[1] banner: w+h or format[] required",
  "specRef": "https://iabtechlab.com/...#322"
}
  • level — error (required-rule violation) / warning (suspicious, not forbidden) / info (note)
  • path — JSON-path to the offending field in your payload
  • msg — human-readable text, localised
  • specRef — direct link to the relevant IAB-spec paragraph

Overall status:

  • errors — at least one level: error
  • warnings — only warnings/info, but not clean
  • clean — nothing found

Quick example

Paste into the BID REQUEST field:

{
  "id": "test-001",
  "imp": [
    { "id": "1", "banner": { "w": 300, "h": 250 }, "bidfloor": 0.5 }
  ],
  "site": { "id": "publisher-abc", "domain": "example.com" },
  "device": { "ua": "Mozilla/5.0", "ip": "1.2.3.4" }
}

Hit analyze stream (or Ctrl+Enter) — you'll see detected version 2.5, a few warnings (no at, tmax, cur), spec-refs to IAB §3.2.1, §3.2.4.

Add into BID RESPONSE:

{
  "id": "test-001",
  "seatbid": [
    { "bid": [
      { "id": "b1", "impid": "1", "price": 1.5,
        "adm": "<html>creative</html>",
        "adomain": ["example.com"], "w": 300, "h": 250 }
    ]}
  ],
  "cur": "USD"
}

Now crosscheck is active — you'll see impid → imp.id match, size match, price > bidfloor check, and the ad preview rendering in the bottom-right card (sandboxed).
