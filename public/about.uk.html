Spyglass — як ним користуватись

Приватний інспектор OpenRTB-трафіку. Вставив JSON, отримав розбір: чи правильно сформований запит/відповідь, які поля обов'язкові, чи проходить crosscheck request↔response, з якою версією oRTB маєш справу, що каже специфікація IAB у конкретному пункті.

Запущено на власному залізі в Києві. Без реклами, без трекерів, без передачі ваших даних третім сторонам.

Що Spyglass вміє

Розбір BidRequest / BidResponse

  • Валідація структури — обов'язкові поля, типи, формати (imp[] непорожній, id рядок, tmax число, cur[] ISO-4217)
  • Підтримка типів реклами — banner, video, native (1.1+1.2), audio. VAST-протоколи у video.protocols, app vs site, rewarded (imp.rwdd)
  • Crosscheck request↔response — bid.impid → imp.id матч, формат креативу збігається з imp.banner/video, ціна не нижче bidfloor, розміри креативу матчать imp.banner.format[]
  • Native crosscheck — звіряє asset-id у запиті з asset-id у відповіді, ловить пропущені/зайві asset-и, невалідний JSON у adm

Detect & inform

  • Автодетект версії — аналізує сигнатурні поля (imp.rwdd, device.sua, regs.gpp_sid, source.schain) і визначає чи це oRTB 2.5 / 2.6 baseline / 2.6-202211 / 2.6-202309 / 3.0. Повертає confidence і список знайдених сигналів.
  • Spec deep-links — кожна знахідка має посилання на конкретний параграф IAB-специфікації (§3.2.10, §5.4)
  • Severity levels — error / warning / info. UI підсвічує іконкою.

Vendor overlays

  • IAB (за замовчуванням) — чиста IAB-специфікація без розширень.
  • Vendor overlays — опціональні overlay-правила для конкретних SSP/DSP (їхні ext.*, custom-макроси, специфічні обмеження). Активуються через ?dialect=<vendor> query-параметр. Публічний UI лишається generic.

Ad preview

Якщо в bid.adm є HTML-креатив — Spyglass рендерить його у sandboxed iframe (sandbox="allow-scripts", без allow-same-origin). Бачиш як виглядає, без ризику для своєї сесії.

Behavior tab — детекція click-skim

Перед тим як HTML креативу почне парситись, Spyglass ін'єктить у iframe маленький instrumentation-probe. Probe огортає window.open, сетери location (assign / replace / href=), HTMLElement.click(), інлайн-сетери onmouseover / onmouseenter / onclick та кожен addEventListener. Він відстежує яка DOM-подія зараз диспатчиться.

Якщо API навігації спрацьовує під час події mouseover / mouseenter / mousemove — це підозра на click-skim: креатив намагається сімулювати клік без того щоб користувач реально клікнув. Якщо взагалі немає події в стеку (наприклад redirect через setTimeout) — це авто-навігація без gesture. Справжні click / pointerdown / mousedown / touchstart події проходять без позначок.

Знахідки з'являються у вкладці Behavior поряд з Validation / Crosscheck / Categories — з URL призначення, методом виклику (window.open, location.href=, …) та подією-тригером. Sandbox і так блокує саму навігацію; probe робить сам факт спроби видимим, щоб його можна було прикріпити до bug-тікета.

Шеринг та ергономіка

  • Посилання — копіює self-contained URL виду spyglass.kyivtech.com.ua/?#req=…&res=…. Payload deflate-стиснутий + base64url-закодований у hash-фрагменті URL, який ніколи не доходить до сервера (zero-knowledge на рівні самої URL). Відкриваєш посилання — обидва поля автоматично заповнюються та аналіз перезапускається.
  • Завантажити бандл — пакує запит + відповідь + знахідки + crosscheck в один JSON-файл (spyglass-{дата}-{6-hex}.json) щоб прикріпити до тікета чи кинути в чат без жонглювання двома великими блобами.
  • Гарячі клавіші — ? показує довідку, Ctrl+Enter запускає аналіз, Ctrl+S зберігає в бібліотеку (потрібен акаунт), Esc закриває будь-яку модалку.

Особистий простір (з акаунтом)

  • Збереження зразків — реальні запити/відповіді з нотатками, групуванням за партнерами
  • Партнери — каталог SSP/DSP/AdNetwork з якими працюєш
  • Zero-knowledge шифрування — все що зберігається, шифрується на твоєму браузері ключем, похідним від твого пароля. Сервер не може розшифрувати твої зразки навіть якщо захоче. При забутому паролі є recovery key — без нього доступ втрачено назавжди.

Чого Spyglass не робить

Обмеження Чому
Не робить реальні bid-и Це інспектор JSON, не RTB-клієнт. Не підключається до жодного exchange.
Не виконує макроси (${AUCTION_PRICE}) у runtime Перевіряємо лише статично — чи макрос валідний, чи правильно записаний.
Не показує реальний рендер креативу з третіх сторін iframe-preview лише локальний adm, без VAST-плеєру і без external-tag fetching.
Не перевіряє повну oRTB 2.6 enforcement Поточна baseline — 2.5 + детект 2.6 сигналів. Strict 2.6 валідація — Phase 2.
Не зберігає зразки якщо ти не залогінений Анонім — все живе тільки у вкладці браузера; reload — і пусто.
Не логує IP/UA при /api/analyze Без auth-у — мінімальний слід (CF Tunnel access-log, не наш бекенд).
Не підтримує English UI Поки тільки українська. Phase 3 roadmap.
Не оптимізує bid strategy / eCPM / yield Це інспектор, не yield manager.
Не парсить TCF/GPP consent strings Запланована окрема вкладка у kyivtech.com.ua/tools/.
Не валідує ads.txt / sellers.json Окремий тулз, не змішаний у oRTB-інспектор.

Які дані Spyglass використовує для аналізу

Локально (in-process, на нашому сервері)

  1. Твій JSON — все що ти вставляєш у BID REQUEST / BID RESPONSE. Парситься, валідується.
  2. IAB OpenRTB 2.5 / 2.6 / 3.0 specs — bundled markdown, для генерації specRef-лінків
  3. Native 1.1 / 1.2 specs — bundled, для asset-validation
  4. VAST 4.x protocol metadata — для video.protocols validation
  5. spec-refs.json — мапа finding-id → IAB spec anchor
  6. messages/uk.json — 71 локалізована повідомлення
  7. Vendor dialect overlays (тільки за ?dialect=<vendor>) — bundled, без жодних викликів до vendor-API

НЕ використовується

  • ❌ Жодних third-party API запитів під час аналізу
  • ❌ Ніяких browser-fingerprinting / tracking pixels
  • ❌ Жодних cookies для анонімних користувачів (тільки session-cookie після login)
  • ❌ Жодних external CDN-ів — все self-hosted
  • ❌ Жодного analytics (Google, Mixpanel, тощо)

Flow аналізу

[твій JSON]
    ↓
parseJSON → validate(structure) → detectVersion → crosscheck
    ↓
findings[]: { id, level, path, params, msg, specRef }
    ↓
сервер форматує локалізований текст і повертає 200 JSON

Все робиться синхронно в одному Node.js процесі, без БД-доступу, без external-fetch, без передачі назовні.

Що зберігається на сервері

Анонімні відвідувачі

Нічого. /api/analyze не пише в БД, не логує payload, не зберігає історію викликів. Cloudflare Tunnel перед нами фіксує сам факт запиту (IP, шлях, статус) — це стандартний access-log, не наш бекенд.

Зареєстровані користувачі

Що Як
email + password hash bcrypt, 12 rounds
Сесія in-memory (TTL 30 днів). Restart серверу — всі вилогінились
Saved samples (bid_req, bid_res) AES-GCM ciphertext. Сервер бачить байти, але не може розшифрувати.
KDF salt + wrapped DEK + IVs Так. Без твого пароля — марно для атакера.
Notes / metadata зразка Те саме шифрування
Партнери (label) Те саме шифрування
email_verified_at timestamp (тільки після verify-email)

Реалізація: пароль на клієнті → PBKDF2 600k SHA-256 → KEK → розгортає збережений DEK → DEK шифрує контент. Сервер не бачить пароля і не може отримати DEK. При зміні пароля DEK перепаковується (не перегенерується), щоб не втратити доступ до старих зразків.

Як читати результат

Кожна знахідка (finding) виглядає так:

{
  "id": "imp.banner.size_required",
  "level": "error",
  "path": "imp[0].banner",
  "params": { "num": 1 },
  "msg": "imp[1] banner: потрібно вказати w+h або format[]",
  "specRef": "https://iabtechlab.com/...#322"
}
  • level — error (порушення обов'язкового правила) / warning (підозріле, не заборонено) / info (нотатка)
  • path — JSON-path до проблемного поля у твоєму payload-і
  • msg — людський текст, локалізований
  • specRef — пряме посилання на параграф IAB-специфікації

Загальний статус:

  • errors — є хоча б одна level: error
  • warnings — лише warnings/info, але не clean
  • clean — нічого не знайдено

Швидкий приклад

Встав у поле BID REQUEST:

{
  "id": "test-001",
  "imp": [
    { "id": "1", "banner": { "w": 300, "h": 250 }, "bidfloor": 0.5 }
  ],
  "site": { "id": "publisher-abc", "domain": "example.com" },
  "device": { "ua": "Mozilla/5.0", "ip": "1.2.3.4" }
}

Натисни analyze stream (або Ctrl+Enter) — побачиш detected version 2.5, кілька warnings (немає at, tmax, cur), spec-refs до IAB §3.2.1, §3.2.4.

Додай у BID RESPONSE:

{
  "id": "test-001",
  "seatbid": [
    { "bid": [
      { "id": "b1", "impid": "1", "price": 1.5,
        "adm": "<html>creative</html>",
        "adomain": ["example.com"], "w": 300, "h": 250 }
    ]}
  ],
  "cur": "USD"
}

Тепер crosscheck активний — побачиш матч impid → imp.id, матч розмірів, перевірку price > bidfloor, і ad preview у нижній правій картці (sandboxed).
