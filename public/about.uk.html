Як користуватися ortbtools

Приватний інспектор OpenRTB-трафіку. Вставив JSON, отримав розбір: чи правильно сформований запит/відповідь, які поля обов'язкові, чи проходить crosscheck request↔response, з якою версією oRTB маєш справу, що каже специфікація IAB у конкретному пункті.

Запущено на власному залізі в Києві. Без реклами, без трекерів, без передачі ваших даних третім сторонам.

Що нового · 2026-05-10

За один день викатили 11 версій (v0.25 → v0.35). Тут — короткий огляд нових функцій. Деталі — у відповідних розділах нижче або у CHANGELOG.md.

🪞 Дзеркало (Mirror) — еталонна пара одним кліком

Натисни дзеркало ↔ у шапці (або hotkey M) — ortbtools згенерує канонічну пару до твого вставленого payload-у. Якщо вставив BidRequest — отримаєш мінімально-валідну BidResponse. Якщо BidResponse — згенерує запит, в якому ця відповідь би пройшла. Результат самоперевіряється через існуючий цикл validate() + crosscheck(), тому завжди чистий. Два режими: minimal (тільки обовʼязкові поля) та best-practice (з recommended: schain, dchain, DSA, GDPR consent, device.sua).

Якщо у тебе заповнені обидві панелі — модалка дзеркала також показує diff: твоя версія vs канонічна, з кольоровим маркуванням (≠ розбіжність, + у канонічній є, − у тебе є зайве). 🔗 share-кнопка генерує permalink що містить пару — корисно для дзвінків з SSP.

Ставити на паузу/продовжити — кнопкою. Клік на рядок завантажує зразок у відповідне поле та закриває модалку. Foundation для майбутньої Stream platform.

🛡 Behavior corpus + Confusion matrix

На вкладці behavior зʼявляється 💾 save as corpus коли є події probe-у і ти залогінений. Зберігаєш event-stream з міткою (legitimate / fraud / ambiguous) — у кабінеті є секція з усіма твоїми entries. Foundation для Chapter B (поведінкова калібровка).

На основі corpus-у автоматично рахується confusion matrix: precision, recall, F1 кожного з 12 детекторів на ТВОЄМУ корпусі. Видно одразу які патерни мають низьку точність (червоний рядок) і потребують tuning, які стабільно ловлять фрод (зелений ≥90%).

🤖 Bid simulator (local LLM 3-strategy)

Кнопка 🤖 simulate у шапці. Локальна gemma4:e2b симулює як 3 типи DSP відповіли б на твій запит: aggressive (max scale, +30-50% над floor), conservative (ROAS guard, +5-15%), quality (premium only, +50-80% або skip). Приватність: LLM бачить тільки 8 полів метаданих (формат, розмір, гео, currency, floor, surface, device-type, auction-type) — значення bid-ів в LLM не потрапляють.

📦 Specimen replay endpoint

Один виклик замість N round-trips. Cap 100 samples на запит. Foundation для CI fixture-grading, archive replay, partner audit pipelines.

Що ortbtools вміє

Розбір BidRequest / BidResponse

  • Валідація структури — обов'язкові поля, типи, формати (imp[] непорожній, id рядок, tmax число, cur[] ISO-4217)
  • Підтримка типів реклами — banner, video, native (1.1+1.2), audio. VAST-протоколи у video.protocols, app vs site, rewarded (imp.rwdd)
  • Crosscheck request↔response — bid.impid → imp.id матч, формат креативу збігається з imp.banner/video, ціна не нижче bidfloor, розміри креативу матчать imp.banner.format[]
  • Native crosscheck — звіряє asset-id у запиті з asset-id у відповіді, ловить пропущені/зайві asset-и, невалідний JSON у adm

Detect & inform

  • Автодетект версії — аналізує сигнатурні поля (imp.rwdd, device.sua, regs.gpp_sid, source.schain) і визначає чи це oRTB 2.5 / 2.6 baseline / 2.6-202211 / 2.6-202309 / 3.0. Повертає confidence і список знайдених сигналів.
  • Spec deep-links — кожна знахідка має посилання на конкретний параграф IAB-специфікації (§3.2.10, §5.4)
  • Severity levels — error / warning / info. UI підсвічує іконкою.

Vendor overlays

  • IAB (за замовчуванням) — чиста IAB-специфікація без розширень.
  • Vendor overlays — опціональні overlay-правила для конкретних SSP/DSP (їхні ext.*, custom-макроси, специфічні обмеження). Активуються через ?dialect=<vendor> query-параметр. Публічний UI лишається generic.

Ad preview

Якщо в bid.adm є HTML-креатив — ortbtools рендерить його у sandboxed iframe (sandbox="allow-scripts", без allow-same-origin). Бачиш як виглядає, без ризику для своєї сесії.

Behavior tab — детекція fraud у креативі

ortbtools рендерить кожен креатив у sandboxed iframe з instrumentation-probe-ом всередині. Probe ловить 11 типів підозрілої поведінки — від невидимих клік-пасток до зависання головного потоку — і вкладка Behavior поряд з Validation / Crosscheck показує структуровані знахідки із severity та URL призначення. Деталі архітектури — у розділі Behavior Analysis & Anti-Fraud нижче.

Шеринг та ергономіка

  • Посилання — копіює self-contained URL виду ortbtools.com/?#req=…&res=…. Payload deflate-стиснутий + base64url-закодований у hash-фрагменті URL, який ніколи не доходить до сервера (zero-knowledge на рівні самої URL). Відкриваєш посилання — обидва поля автоматично заповнюються та аналіз перезапускається.
  • Завантажити бандл — пакує запит + відповідь + знахідки + crosscheck в один JSON-файл (spyglass-{дата}-{6-hex}.json) щоб прикріпити до тікета чи кинути в чат без жонглювання двома великими блобами.
  • Гарячі клавіші — ? показує довідку, Ctrl+Enter запускає аналіз, Ctrl+S зберігає в бібліотеку (потрібен акаунт), Esc закриває будь-яку модалку.

Особистий простір (з акаунтом)

  • Збереження зразків — реальні запити/відповіді з нотатками, групуванням за партнерами
  • Партнери — каталог SSP/DSP/AdNetwork з якими працюєш
  • Zero-knowledge шифрування — все що зберігається, шифрується на твоєму браузері ключем, похідним від твого пароля. Сервер не може розшифрувати твої зразки навіть якщо захоче. При забутому паролі є recovery key — без нього доступ втрачено назавжди.

Підтримувані формати реклами

ortbtools розпізнає формат payload-у і вішає кольорові chip-и (синій — формат, зелений — контекст, бурштиновий — протокол) у лівій панелі підсумку:

  • Banner (display) — desktop і mobile, multi-size banner.format[], MRAID-API
  • Video — instream / outstream, VAST 2 / 3 / 4 (включно з wrapper-ами), CTV (Connected TV), rewarded
  • Native — IAB Native 1.1 + 1.2, asset-id crosscheck (title, image, icon, sponsored, body)
  • Audio — DAAST + VAST-audio, podcast inventory
  • In-App — мобільні bundles, MRAID, OMID
  • DOOH (Digital Out-of-Home) — billboards, transit, qty.multiplier
  • JsonFeed — push-нотифікації, popunders, in-page widgets (vendor-specific JSON-feed формати)

ortbtools Intelligence — локальний AI-асистент

Якщо адмін ortbtools підключив локальний Ollama з моделлю gemma4:e2b — ти отримуєш AI-помічника, який вміє називати кластери нових полів у Discovery-режимі та пояснювати призначення окремих ext.*-полів. Все працює на тій самій машині, що ortbtools, без публічного інтернету:

  • 100% локально — модель крутиться у Docker-контейнері поряд із ortbtools. Жоден токен не виходить за межі LAN.
  • Тільки шляхи полів — у промпт ніколи не потрапляють значення з твого bid-у. Передаються лише назви полів і char-class підказки (паралельно з PII-фільтром).
  • On-demand — AI не аналізує нічого автоматично. Натискаєш «🤖 Suggest» у Dialect Builder — отримуєш відповідь. Не натискаєш — нічого не відбувається.
  • Knowledge Base як контекст — модель отримує 1-2 еталонні приклади поля з нашої вбудованої бази формату (push, banner, native…), щоб імена кластерів були заземлені на реальний AdTech-словник, а не на пріори.
  • Fail-open — якщо Ollama вимкнено чи ламається, AI-кнопки тихо ховаються. Решта ortbtools працює як завжди. Без AI втрачаєш лише авто-іменування кластерів — все інше працює.

Safe Demo Mode — блюр для скріншотів

Додаєш ?demo=safe до URL — і ortbtools:

  • Накладає CSS-blur на вікно прев'ю креативу (видно силует, не видно змісту)
  • Маскує доменні імена у summary-панелі (***rt.com замість реального домену)
  • Не торкається валідації — двигун працює на 100%, ховається лише візуальний шар

Зручно для скріншотів у документацію, презентацій клієнтам, або коли треба показати тулз без розкриття конкретних адверт-партнерів — особливо актуально для adult / casino / gambling трафіку, де навіть силует логотипа це лік.

Як працює аналіз — три шари

ortbtools дивиться на твій payload одночасно з трьох ракурсів. Кожен шар відповідає на своє питання; разом вони дають повну картину.

Шар 1 — Static (схема)

«Чи правильно сформовано JSON?» Парсимо payload, валідуємо проти IAB-специ (oRTB 2.5 / 2.6 / 3.0, Native 1.x, VAST-протоколи), перевіряємо обов'язкові поля, типи, формати. Результат — список знахідок із severity (error / warning / info), JSON-path до місця помилки і deep-link на конкретний параграф IAB-документації. Це найшвидший шар — мілісекунди на pure-JS правилах, без жодних побічних викликів.

Шар 2 — Behavior (sandbox iframe)

«А що цей креатив реально робить у браузері?» Рендеримо bid.adm у sandboxed iframe (sandbox="allow-scripts", без allow-same-origin) з instrumentation-probe-ом всередині. Probe ловить підозрілу поведінку: invisible click-traps, frame-bust attempts, frozen-thread heartbeat-и, permission abuse (Notification, geolocation, getUserMedia), miner-сигнатури в коді, click-skim probe-и (фейкові клік-події що випереджають реальний клік користувача). Engine видає структуровані знахідки — той самий шейп, що Static, плюс severity та URL призначення. Деталі — у розділі Behavior Analysis & Anti-Fraud >.

Шар 3 — Knowledge Base (вбудована енциклопедія)

«Я вже бачив такий формат?» У packages/core/knowledge_base/ лежить курована колекція еталонних oRTB / JsonFeed payload-ів — banner / video / audio / native / push / pops / inpage / inapp / dooh / ctv. Format-detect-двигун тегує твій payload по цій базі (банер VAST в InApp на Android — типовий приклад) і ці теги служать двом цілям одночасно: вони рендеряться як chip-и в UI, і вони підкидаються в промпт локального LLM як few-shot context, щоб AI відповідав словником реального ринку, а не пріорів. База license-clean (синтезовані семпли + Apache-2.0 / MIT з Prebid.js та IAB), без жодних реальних bid-стрімів.

Behavior Analysis & Anti-Fraud

JSON-валідація відповідає на питання «чи правильно сформовано запит/відповідь?». Behavior-двигун відповідає на принципово інше: «а що цей креатив реально робить у браузері користувача?». Це другий шар — runtime-аналіз поведінки HTML/JS, який не видно зі звичайного аналізу payload-у.

Як це працює

ortbtools рендерить креатив у sandboxed iframe з sandbox="allow-scripts" — без allow-same-origin, без allow-top-navigation, без allow-popups. Цього достатньо щоб заблокувати більшість зловмисних дій. Але блокування ≠ видимість: креатив, який намагається виконати fraud, проти заблокованого sandbox-а просто мовчить — користувач не бачить що щось було не так.

Тому перед HTML креативу в iframe-у запускається creative-probe (~830 рядків JavaScript, 15 instrumentation hooks). Probe огортає навігаційні API (window.open, Location.assign / replace / href=), програмні кліки (HTMLElement.click, HTMLFormElement.submit), прослуховувачі подій, плюс підключає PerformanceObserver(longtask), PerformanceObserver(resource) та 1Hz heartbeat. Кожна підозріла дія у момент виявлення — postMessage до батьківського вікна.

Probe також вимірює реальні розміри елементів через getBoundingClientRect() + getComputedStyle() у момент кліку — це дає змогу детектити невидимі шари-пастки (opacity < 0.05 на елементі, що покриває > 50% viewport-у), занадто малі close-кнопки (< 44×44 IAB) та центровані синтетичні кліки від pixelbot-ів.

Батьківське вікно збирає події і POST-ить їх на /api/analyze-behavior, де engine у packages/core/behavior/ — pure-data функція — матчить шаблони і видає structured findings із severity (error / warning / info), URL призначення, метаданими (gesture lineage, threshold breached, тощо). Engine ізоморфний (browser-and-Node compatible), але в цьому деплої виконується на сервері за цим endpoint-ом — той самий код також обробляє replay архівних специменів на сервері.

Чотири вектори захисту

1. Misclick traps (Клік-пастки)

Креатив монтує невидимий елемент (opacity=0, transparent background) розміром на весь viewport, поверх видимого контенту. Будь-який клік користувача потрапляє НЕ туди де він збирався — а в пастку, яка реєструє клік як подію конверсії або редиректить на лендінг. Probe вимірює geometry таргета та computed style: якщо bounding rect покриває > 50% viewport-у і opacity < 0.05 → behavior.trap.invisible_overlay (severity ERROR).

2. Bot Patterns (Боти)

Pixelbot-скрипти зчитують центр кнопки через getBoundingClientRect() і диспатчать click у точно цю координату — але event.isTrusted у синтезованому кліку завжди false (виставити true можна тільки через WebDriver / CDP, до яких креатив у sandbox-і не має доступу). Probe ловить три патерни:

  • center-synth — відстань < 0.5px від центра + isTrusted=false (behavior.bot.center_synth, ERROR)
  • click-burst — 3+ кліків у 200мс sliding-window-і (behavior.bot.click_burst, ERROR — людина так не клацає)
  • phantom-click — клік без жодного попереднього mousemove чи touchstart з моменту завантаження креативу (behavior.bot.phantom_click, WARNING — типово для headless-браузерів)

3. Malicious Ads — Frame Busting & Auto-Redirects

Найпідступніший вектор: креатив намагається вирватися з sandboxed iframe-у на top frame. Через <a target="_top"> анкори, <form target="_top"> сабміт або auto-redirect через setTimeout. Sandbox блокує саму навігацію (немає allow-top-navigation), але probe бачить спробу. Engine трекає історію user-gesture через navigator.userActivation + ручний _lastGestureAt snapshot і розрізняє чотири підпатерни:

  • frame_bust_anchor — клік на <a target="_top">; ERROR без gesture lineage, WARNING якщо в межах 500мс gesture-grace
  • frame_bust_form — submit форми з target="_top"; завжди ERROR (форми у банері з top-навігацією не мають легітимного use case-у)
  • auto_redirect — навігація з порожнім event-stack-ом і без recent gesture (ERROR)
  • late_redirect — gesture був, але навігація через setTimeout > 500мс пізніше (WARNING — класичний клоакінг: користувач клацнув щось видиме, потім setTimeout провів реальну навігацію поза gesture-lineage)

4. Heavy Ads (Зажерливі креативи)

Креатив, який блокує main thread надмірно або тягне мегабайти. Probe використовує PerformanceObserver(longtask) для акумуляції CPU-часу і PerformanceObserver(resource) для transferSize. Пороги мирорять Chrome Heavy Ad Intervention:

  • heavy_ad_cpu — > 60с cumulative blocking АБО > 4с у будь-якому 30-секундному вікні (ERROR)
  • heavy_ad_network — > 4 MB сабресурсів cumulative (ERROR; IAB display-ad weight cap)
  • frozen_thread — окремий механізм через parent watchdog: probe шле 1Hz heartbeat, parent моніторить лаг — якщо > 6с без сигналу, генерується синтетична подія (probe не може репортити власне зависання — нескінченний цикл блокує і setInterval і postMessage, тому детекція повинна жити поза iframe-ом). ERROR.

Probe працює client-side у пісочниці iframe; зібрані поведінкові події (разом із розміткою креативу) надсилаються на /api/analyze-behavior і оцінюються на сервері, який нічого не зберігає. Engine у packages/core/behavior/ — pure-data функція, що лише повертає знахідки. Sandbox блокує саму fraud-дію; ми робимо факт спроби видимим — щоб ти міг прикріпити до bug-тікета і повернутися до постачальника креативу зі screenshot-ом.

Які дані ortbtools використовує для аналізу

Локально (in-process, на нашому сервері)

  1. Твій JSON — все що ти вставляєш у BID REQUEST / BID RESPONSE. Парситься, валідується.
  2. IAB OpenRTB 2.5 / 2.6 / 3.0 specs — bundled markdown, для генерації specRef-лінків
  3. Native 1.1 / 1.2 specs — bundled, для asset-validation
  4. VAST 4.x protocol metadata — для video.protocols validation
  5. spec-refs.json — мапа finding-id → IAB spec anchor
  6. messages/{uk,en,ru}.json — локалізовані повідомлення
  7. Vendor dialect overlays (тільки за ?dialect=<vendor>) — bundled, без жодних викликів до vendor-API

НЕ використовується

  • ❌ Жодних third-party API запитів під час аналізу
  • ❌ Ніяких browser-fingerprinting / tracking pixels
  • ❌ Жодних cookies для анонімних користувачів (тільки session-cookie після login)
  • ❌ Жодних external CDN-ів — все self-hosted
  • ❌ Жодного analytics (Google, Mixpanel, тощо)

Flow аналізу

[твій JSON]
    ↓
parseJSON → validate(structure) → detectVersion → crosscheck
    ↓
findings[]: { id, level, path, params, msg, specRef }
    ↓
сервер форматує локалізований текст і повертає 200 JSON

Сама валідація відбувається синхронно в одному Node.js процесі — жоден зовнішній сервіс не залучається для оцінки твого bid. Після відповіді записуються лише похідні метадані (формат, версія oRTB, кількість знахідок) у нашу аналітику та логи запитів; тіла request і response не записуються нікуди.

Що зберігається на сервері

Анонімні відвідувачі

Твій payload — нічого. Тіла request і response валідуються в памʼяті й відкидаються; /api/analyze ніколи не записує їх у базу чи лог. Ми зберігаємо знеособлену аналітику (визначений формат, версію oRTB, кількість знахідок) та операційний лог запитів, що фіксує метадані запиту — час, метод, шлях, статус і твою IP-адресу (кожну помилку плюс вибірку успішних викликів) — але ніколи не payload. Сам Cloudflare Tunnel перед нами теж веде стандартний access-log.

Зареєстровані користувачі

Що Як
email + password hash bcrypt, 12 rounds
Сесія Токен + термін дії зберігаються в SQLite, відновлюються при старті (TTL 30 днів). Рядок також зберігає твій IP і браузерний user-agent.
Saved samples (bid_req, bid_res) AES-GCM ciphertext. Сервер бачить байти, але не може розшифрувати.
KDF salt + wrapped DEK + IVs Так. Без твого пароля — марно для атакера.
Notes / metadata зразка Те саме шифрування
Партнери (label) Те саме шифрування
email_verified_at timestamp (тільки після verify-email)

Реалізація: у браузері пароль проходить через PBKDF2 600k SHA-256 → KEK → розгортає збережений DEK → DEK шифрує контент. Пароль також надсилається на сервер через HTTPS для перевірки логіна через bcrypt — сервер зберігає лише bcrypt-хеш, ніколи не відкритий текст, а KEK і розгорнутий DEK ніколи не залишають браузер. Тому сервер не може розшифрувати твої збережені зразки. При зміні пароля DEK перепаковується (не перегенерується), щоб не втратити доступ до старих зразків.

Як читати результат

Кожна знахідка (finding) виглядає так:

{
  "id": "imp.banner.size_required",
  "level": "error",
  "path": "imp[0].banner",
  "params": { "num": 1 },
  "msg": "imp[1] banner: потрібно вказати w+h або format[]",
  "specRef": "https://iabtechlab.com/...#322"
}
  • level — error (порушення обов'язкового правила) / warning (підозріле, не заборонено) / info (нотатка)
  • path — JSON-path до проблемного поля у твоєму payload-і
  • msg — людський текст, локалізований
  • specRef — пряме посилання на параграф IAB-специфікації

Загальний статус:

  • errors — є хоча б одна level: error
  • warnings — лише warnings/info, але не clean
  • clean — нічого не знайдено

Швидкий приклад

Встав у поле BID REQUEST:

{
  "id": "test-001",
  "imp": [
    { "id": "1", "banner": { "w": 300, "h": 250 }, "bidfloor": 0.5 }
  ],
  "site": { "id": "publisher-abc", "domain": "example.com" },
  "device": { "ua": "Mozilla/5.0", "ip": "1.2.3.4" }
}

Натисни analyze stream (або Ctrl+Enter) — побачиш detected version 2.5, кілька warnings (немає at, tmax, cur), spec-refs до IAB §3.2.1, §3.2.4.

Додай у BID RESPONSE:

{
  "id": "test-001",
  "seatbid": [
    { "bid": [
      { "id": "b1", "impid": "1", "price": 1.5,
        "adm": "<html>creative</html>",
        "adomain": ["example.com"], "w": 300, "h": 250 }
    ]}
  ],
  "cur": "USD"
}

Тепер crosscheck активний — побачиш матч impid → imp.id, матч розмірів, перевірку price > bidfloor, і ad preview у нижній правій картці (sandboxed).
