Как пользоваться ortbtools

Приватный инспектор OpenRTB-трафика. Вставь JSON, получи разбор: правильно ли сформирован запрос/ответ, какие поля обязательные, проходит ли crosscheck request↔response, с какой версией oRTB имеешь дело, что говорит спецификация IAB по конкретному пункту.

Self-hosted на собственном железе в Киеве. Без рекламы, без трекеров, без передачи ваших данных третьим сторонам.

Что нового · 2026-05-10

За один день выкатили 11 версий (v0.25 → v0.35). Краткий обзор новых функций — детали в соответствующих разделах ниже или в CHANGELOG.md.

🪞 Зеркало (Mirror) — эталонная пара одним кликом

Жми зеркало ↔ в шапке (или hotkey M) — ortbtools сгенерирует канонический counterpart твоего payload-а. Если вставил BidRequest — получишь минимально-валидный BidResponse. Если BidResponse — сгенерирует запрос, в котором этот ответ прошёл бы. Результат самопроверяется через существующий цикл validate() + crosscheck(), поэтому всегда чистый. Два режима: minimal (только обязательные поля) и best-practice (с recommended: schain, dchain, DSA, GDPR consent, device.sua).

Если у тебя заполнены оба поля — модалка зеркала также показывает diff: твоя версия vs каноническая, с цветовой маркировкой (≠ различие, + в канонической есть, − у тебя есть лишнее). 🔗 share-кнопка генерирует permalink с парой — полезно для звонков с SSP.

📡 Live RTB stream

Кнопка 📡 live открывает модалку со стримом синтетических RTB-образцов в реальном времени. Каждая строка: timestamp, тип (REQ/RES), источник (имя файла), опционально размер баннера. Пауза/продолжить — кнопкой. Клик на строке загружает образец в соответствующее поле и закрывает модалку. Foundation для будущей Stream platform.

👁 Детали находок (Finding details)

Каждая находка валидатора теперь имеет ▾ разворачивалку. Клик открывает панель с: JSON-путём, твоим текущим значением в этом пути (вытаскиваем из вставленного), серьёзностью с человеческой расшифровкой ("error → биржи отклонят"; "warning → fill-rate упадёт"; "info → best-practice"), ссылкой на спецификацию и canonical ID правила.

🛡 Behavior corpus + Confusion matrix

На вкладке behavior появляется 💾 save as corpus когда есть события probe-а и ты залогинен. Сохраняешь event-stream с меткой (legitimate / fraud / ambiguous) — в кабинете есть секция со всеми твоими entries. Foundation для Chapter B (поведенческая калибровка).

На основе corpus-а автоматически считается confusion matrix: precision, recall, F1 каждого из 12 детекторов на ТВОЁМ корпусе. Видно сразу какие паттерны имеют низкую точность (красная строка) и нуждаются в tuning, какие стабильно ловят фрод (зелёный ≥90%).

🤖 Bid simulator (local LLM 3-strategy)

Кнопка 🤖 simulate в шапке. Локальная gemma4:e2b симулирует как 3 типа DSP ответили бы на твой запрос: aggressive (max scale, +30-50% над floor), conservative (ROAS guard, +5-15%), quality (premium only, +50-80% или skip). Приватность: LLM видит только 8 полей метаданных (формат, размер, гео, currency, floor, surface, device-type, auction-type) — значения bid-ов в LLM не попадают.

📦 Specimen replay endpoint

POST /api/v1/replay принимает массив samples, прогоняет валидатор + crosscheck + behavior на каждом, возвращает per-sample результаты + агрегат (statusCounts, topFindings, totalFindings). Один вызов вместо N round-trips. Cap 100 samples на запрос. Foundation для CI fixture-grading, archive replay, partner audit pipelines.

👤 Кабинет — новая структура с 7 разделами

Cabinet (/ru/account) переосмыслен: боковая навигация (sticky sidebar) с 7 разделами — Профиль · Библиотека · Активность · Behavior corpus · Настройки · Безопасность · Danger zone. Scroll-spy подсвечивает активную секцию при скролле. URL hash обновляется для глубоких ссылок: /ru/account#corpus открывает сразу корпус. Mobile (≤880px) — sidebar сворачивается в горизонтальные tab-чипы.

Что ortbtools умеет

Разбор BidRequest / BidResponse

  • Валидация структуры — обязательные поля, типы, форматы (imp[] непустой, id строка, tmax число, cur[] ISO-4217)
  • Поддержка типов рекламы — banner, video, native (1.1 + 1.2), audio. VAST-протоколы в video.protocols, app vs site, rewarded (imp.rwdd)
  • Crosscheck request↔response — bid.impid → imp.id match, формат креатива совпадает с imp.banner/video, цена ≥ bidfloor, размер креатива совпадает с imp.banner.format[]
  • Native crosscheck — сверяет asset-id из запроса с asset-id из ответа, ловит пропущенные/лишние asset-ы, невалидный JSON в adm

Detect & inform

  • Авто-детект версии — анализирует сигнатурные поля (imp.rwdd, device.sua, regs.gpp_sid, source.schain) и определяет, oRTB 2.5 / 2.6 baseline / 2.6-202211 / 2.6-202309 / 3.0 это. Возвращает confidence и список найденных сигналов.
  • Spec deep-links — каждая находка имеет ссылку на конкретный параграф IAB-спецификации (§3.2.10, §5.4)
  • Severity levels — error / warning / info. UI подсвечивает иконкой.

Vendor overlays

  • IAB (по умолчанию) — чистая IAB-спецификация без расширений.
  • Vendor overlays — опциональные overlay-правила для конкретных SSP/DSP (их ext.*, custom-макросы, специфичные ограничения). Активируются через ?dialect=<vendor> query-параметр. Публичный UI остаётся стандартным.

Ad preview

Если в bid.adm есть HTML-креатив — ortbtools рендерит его в sandboxed iframe (sandbox="allow-scripts", без allow-same-origin). Видишь как выглядит, без риска для своей сессии.

Behavior tab — детекция fraud в креативе

ortbtools рендерит каждый креатив в sandboxed iframe с instrumentation-probe-ом внутри. Probe ловит 11 типов подозрительного поведения — от невидимых клик-ловушек до зависания главного потока — и вкладка Behavior рядом с Validation / Crosscheck показывает структурированные находки с severity и URL назначения. Архитектурный deep-dive — в разделе Behavior Analysis & Anti-Fraud ниже.

Шеринг и эргономика

  • Ссылка — копирует self-contained URL вида ortbtools.com/?#req=…&res=…. Payload deflate-сжатый + base64url-закодированный в hash-фрагменте URL, который никогда не доходит до сервера (zero-knowledge на уровне самой URL). Открываешь ссылку — оба поля автоматически заполняются и анализ перезапускается.
  • Скачать бандл — пакует запрос + ответ + находки + crosscheck в один JSON-файл (spyglass-{дата}-{6-hex}.json) чтобы прикрепить к тикету или кинуть в чат без жонглирования двумя большими блобами.
  • Горячие клавиши — ? показывает справку, Ctrl+Enter запускает анализ, Ctrl+S сохраняет в библиотеку (нужен аккаунт), Esc закрывает любую модалку.

Личное пространство (с аккаунтом)

  • Сохранение образцов — реальные запросы/ответы с заметками, группировкой по партнёрам
  • Партнёры — каталог SSP/DSP/AdNetwork с которыми работаешь
  • Zero-knowledge шифрование — всё что сохраняется, шифруется в твоём браузере ключом, производным от твоего пароля. Сервер не может расшифровать твои образцы даже если захочет. Есть recovery key для случая забытого пароля — без него доступ потерян навсегда.

Поддерживаемые форматы рекламы

ortbtools распознаёт формат payload-а и рисует цветные chip-ы (синий — формат, зелёный — контекст, янтарный — протокол) в левой панели сводки:

  • Banner (display) — desktop и mobile, multi-size banner.format[], MRAID API
  • Video — instream / outstream, VAST 2 / 3 / 4 (включая wrapper-ы), CTV (Connected TV), rewarded
  • Native — IAB Native 1.1 + 1.2, asset-id crosscheck (title, image, icon, sponsored, body)
  • Audio — DAAST + VAST-audio, podcast inventory
  • In-App — мобильные bundles, MRAID, OMID
  • DOOH (Digital Out-of-Home) — billboards, transit, qty.multiplier
  • JsonFeed — push-нотификации, popunders, in-page widgets (vendor-specific JSON-feed форматы)

ortbtools Intelligence — локальный AI-ассистент

Если админ ortbtools подключил локальный Ollama с моделью gemma4:e2b — ты получаешь AI-помощника, который умеет называть кластеры новых полей в Discovery-режиме и пояснять назначение отдельных ext.*-полей. Всё работает на той же машине, что ortbtools, без публичного интернета:

  • 100% локально — модель крутится в Docker-контейнере рядом со ortbtools. Ни один токен не покидает LAN.
  • Только пути полей — в промпт никогда не попадают значения из твоего bid-а. Передаются только имена полей и char-class подсказки (плюс PII-фильтр).
  • On-demand — AI не анализирует ничего автоматически. Нажимаешь «🤖 Suggest» в Dialect Builder — получаешь ответ. Не нажимаешь — ничего не происходит.
  • Knowledge Base как контекст — модель получает 1-2 эталонных примера полей из нашей встроенной базы формата (push, banner, native…), чтобы имена кластеров были заземлены на реальный AdTech-словарь, а не на приоры.
  • Fail-open — если Ollama выключен или сломался, AI-кнопки тихо прячутся. Остальной ortbtools работает как обычно. Без AI теряешь только авто-именование кластеров — всё остальное работает.

Safe Demo Mode — блюр для скриншотов

Добавляешь ?demo=safe к URL — и ortbtools:

  • Накладывает CSS-blur на окно превью креатива (виден силуэт, не виден контент)
  • Маскирует доменные имена в summary-панели (***rt.com вместо реального домена)
  • Не трогает валидацию — движок работает на 100%, прячется только визуальный слой

Удобно для скриншотов в документацию, презентаций клиентам, или когда нужно показать тулз без раскрытия конкретных адверт-партнёров — особенно актуально для adult / casino / gambling трафика, где даже силуэт логотипа это утечка.

Как работает анализ — три слоя

ortbtools смотрит на твой payload одновременно с трёх ракурсов. Каждый слой отвечает на свой вопрос; вместе они дают полную картину.

Слой 1 — Static (схема)

«Правильно ли сформирован JSON?» Парсим payload, валидируем против IAB-спеки (oRTB 2.5 / 2.6 / 3.0, Native 1.x, VAST-протоколы), проверяем обязательные поля, типы, форматы. Результат — список находок с severity (error / warning / info), JSON-path до места ошибки и deep-link на конкретный параграф IAB-документации. Это самый быстрый слой — миллисекунды на pure-JS правилах, без побочных вызовов.

Слой 2 — Behavior (sandbox iframe)

«А что этот креатив реально делает в браузере?» Рендерим bid.adm в sandboxed iframe (sandbox="allow-scripts", без allow-same-origin) с instrumentation-probe-ом внутри. Probe ловит подозрительное поведение: invisible click-traps, frame-bust attempts, frozen-thread heartbeat-ы, permission abuse (Notification, geolocation, getUserMedia), miner-сигнатуры в коде, click-skim probe-ы (фейковые клик-события, опережающие реальный клик пользователя). Engine выдаёт structured findings — тот же шейп, что Static, плюс severity и URL назначения. Детали — в разделе Behavior Analysis & Anti-Fraud >.

Слой 3 — Knowledge Base (встроенная энциклопедия)

«Я уже видел такой формат?» В packages/core/knowledge_base/ лежит курируемая коллекция эталонных oRTB / JsonFeed payload-ов — banner / video / audio / native / push / pops / inpage / inapp / dooh / ctv. Format-detect-движок тегирует твой payload по этой базе (баннер VAST в InApp на Android — типичный пример), и эти теги служат двум целям одновременно: они рендерятся как chip-ы в UI, и они подкидываются в промпт локального LLM как few-shot context, чтобы AI отвечал словарём реального рынка, а не приоров. База license-clean (синтезированные семплы + Apache-2.0 / MIT из Prebid.js и IAB), без каких-либо реальных bid-стримов.

Behavior Analysis & Anti-Fraud

JSON-валидация отвечает на вопрос «правильно ли сформирован запрос/ответ?». Behavior-движок отвечает на принципиально другой: «а что этот креатив реально делает в браузере пользователя?». Это второй слой — runtime-анализ поведения HTML/JS, который не виден из обычного анализа payload-а.

Как это работает

ortbtools рендерит креатив в sandboxed iframe c sandbox="allow-scripts" — без allow-same-origin, без allow-top-navigation, без allow-popups. Этого достаточно чтобы заблокировать большинство вредоносных действий. Но блокировка ≠ видимость: креатив, который пытается выполнить fraud, против заблокированного sandbox-а просто молчит — пользователь не видит что что-то пошло не так.

Поэтому перед HTML креатива в iframe-е запускается creative-probe (~830 строк JavaScript, 15 instrumentation hooks). Probe оборачивает навигационные API (window.open, Location.assign / replace / href=), программные клики (HTMLElement.click, HTMLFormElement.submit), слушатели событий, плюс подключает PerformanceObserver(longtask), PerformanceObserver(resource) и 1Hz heartbeat. Каждое подозрительное действие в момент обнаружения — postMessage в родительское окно.

Probe также измеряет реальные размеры элементов через getBoundingClientRect() + getComputedStyle() в момент клика — это позволяет детектить невидимые слои-ловушки (opacity < 0.05 на элементе, перекрывающем > 50% вьюпорта), слишком маленькие close-кнопки (< 44×44 IAB) и центрированные синтетические клики от pixelbot-ов.

Родительское окно прогоняет события через engine в packages/core/behavior/ — pure-data функцию, которая матчит шаблоны и выдаёт структурированные находки с severity (error / warning / info), URL назначения, метаданными (gesture lineage, threshold breached, и т.п.). Engine browser-and-Node compatible: запускается в браузере через /api/analyze-behavior и на сервере для replay архивных специменов.

Четыре вектора защиты

1. Misclick traps (Клик-ловушки)

Креатив монтирует невидимый элемент (opacity=0, transparent background) размером на весь вьюпорт, поверх видимого контента. Любой клик пользователя попадает НЕ туда куда он собирался — а в ловушку, которая регистрирует клик как событие конверсии или редиректит на лендинг. Probe измеряет geometry таргета и computed style: если bounding rect перекрывает > 50% вьюпорта и opacity < 0.05 → behavior.trap.invisible_overlay (severity ERROR).

2. Bot Patterns (Боты)

Pixelbot-скрипты считывают центр кнопки через getBoundingClientRect() и диспатчат click в точно эту координату — но event.isTrusted в синтезированном клике всегда false (выставить true можно только через WebDriver / CDP, к которым креатив в sandbox-е не имеет доступа). Probe ловит три паттерна:

  • center-synth — расстояние < 0.5px от центра + isTrusted=false (behavior.bot.center_synth, ERROR)
  • click-burst — 3+ кликов в 200мс sliding-window-е (behavior.bot.click_burst, ERROR — человек так не кликает)
  • phantom-click — клик без единого предшествующего mousemove или touchstart с момента загрузки креатива (behavior.bot.phantom_click, WARNING — типично для headless-браузеров)

3. Malicious Ads — Frame Busting и Auto-Redirects

Самый коварный вектор: креатив пытается вырваться из sandboxed iframe-а на top frame. Через <a target="_top"> анкоры, <form target="_top"> сабмит или auto-redirect через setTimeout. Sandbox блокирует саму навигацию (нет allow-top-navigation), но probe видит попытку. Engine трекает историю user-gesture через navigator.userActivation + ручной _lastGestureAt snapshot и разделяет на четыре подпаттерна:

  • frame_bust_anchor — клик на <a target="_top">; ERROR без gesture lineage, WARNING если в пределах 500мс gesture-grace
  • frame_bust_form — submit формы с target="_top"; всегда ERROR (формы в баннере с top-навигацией не имеют легитимного use case-а)
  • auto_redirect — навигация с пустым event-stack-ом и без recent gesture (ERROR)
  • late_redirect — gesture был, но навигация сработала через setTimeout > 500мс позже (WARNING — классический клоакинг: пользователь кликнул что-то видимое, потом setTimeout провёл реальную навигацию вне gesture-lineage)

4. Heavy Ads (Прожорливые креативы)

Креатив, который блокирует main thread чрезмерно или тянет мегабайты. Probe использует PerformanceObserver(longtask) для аккумуляции CPU-времени и PerformanceObserver(resource) для transferSize. Пороги мирорят Chrome Heavy Ad Intervention:

  • heavy_ad_cpu — > 60с cumulative blocking ИЛИ > 4с в любом 30-секундном окне (ERROR)
  • heavy_ad_network — > 4 MB сабресурсов cumulative (ERROR; IAB display-ad weight cap)
  • frozen_thread — отдельный механизм через parent watchdog: probe шлёт 1Hz heartbeat, parent мониторит лаг — если > 6с без сигнала, генерируется синтетическое событие (probe не может репортить собственное зависание — бесконечный цикл блокирует и setInterval, и postMessage, поэтому детекция должна жить вне iframe-а). ERROR.

Всё это работает client-side, ноль нагрузки на бекенд. Engine в packages/core/behavior/ — pure-data функция; находки рендерятся во вкладке Behavior рядом с Validation / Crosscheck. Sandbox блокирует само fraud-действие; мы делаем факт попытки видимым — чтобы можно было прикрепить к bug-тикету и вернуться к поставщику креатива со screenshot-ом.

Что ortbtools не делает

Ограничение Почему
Не делает реальные bid-ы Это инспектор JSON, не RTB-клиент. Не подключается ни к одному exchange.
Не исправляем JSON автоматически Мы указываем на ошибки + ссылку на параграф спецификации, но payload не редактируем. JSON-fix — твоя зона ответственности.
Не анализируем сами пиксели видео / картинок Sandbox-iframe ловит HTML/JS-поведение (клики, redirect, permission requests, малвер-паттерны). Что изображает сама картинка или видеокадр — не смотрим.
AI не угадывает логику неизвестных полей без твоей команды Suggest срабатывает только на нажатие 🤖 в Dialect Builder. Никакого background-анализа, никакого «AI knows best» — ты контролируешь каждый вызов.
Не выполняет макросы (${AUCTION_PRICE}) в runtime Проверяем только статически — валидный ли макрос, правильно ли записан.
Не показывает реальный рендер креативов от третьих сторон iframe-preview только локальный adm, без VAST-плеера и без external-tag fetching.
Не проверяет полную oRTB 2.6 enforcement Текущий baseline — 2.5 + детект 2.6 сигналов. Strict 2.6 валидация — Phase 2.
Не сохраняет образцы если ты не залогинен Аноним — всё живёт только во вкладке браузера; reload — и пусто.
Не логирует IP/UA на /api/analyze Без auth-а — минимальный след (CF Tunnel access-log, не наш бекенд).
Не оптимизирует bid strategy / eCPM / yield Это инспектор, не yield manager.
Не парсит TCF/GPP consent strings Запланирован отдельный инструмент на ortbtools.com.
Не валидирует ads.txt / sellers.json Отдельный тул, не смешан в oRTB-инспектор.

Какие данные ortbtools использует для анализа

Локально (in-process, на нашем сервере)

  1. Твой JSON — всё что ты вставляешь в BID REQUEST / BID RESPONSE. Парсится, валидируется.
  2. IAB OpenRTB 2.5 / 2.6 / 3.0 specs — bundled markdown, используется для генерации specRef-ссылок
  3. Native 1.1 / 1.2 specs — bundled, используется для asset-валидации
  4. VAST 4.x protocol metadata — используется для video.protocols валидации
  5. spec-refs.json — map finding-id → IAB spec anchor
  6. messages/{uk,en,ru}.json — локализованные сообщения о находках
  7. Vendor dialect overlays (only with ?dialect=<vendor>) — bundled, без вызовов vendor-API

НЕ используется

  • ❌ Никаких third-party API запросов во время анализа
  • ❌ Никакого browser-fingerprinting / tracking pixels
  • ❌ Никаких cookies для анонимных пользователей (только session-cookie после login)
  • ❌ Никаких external CDN — всё self-hosted
  • ❌ Никакой analytics (Google, Mixpanel, и т.д.)

Flow анализа

[твой JSON]
    ↓
parseJSON → validate(structure) → detectVersion → crosscheck
    ↓
findings[]: { id, level, path, params, msg, specRef }
    ↓
сервер форматирует локализованный текст и возвращает 200 JSON

Всё делается синхронно в одном Node.js процессе, без БД-доступа, без external-fetch, без передачи наружу.

Что хранится на сервере

Анонимные посетители

Ничего. /api/analyze не пишет в БД, не логирует payload, не сохраняет историю вызовов. Cloudflare Tunnel перед нами фиксирует сам факт запроса (IP, путь, статус) — это стандартный access-log, не наш бекенд.

Зарегистрированные пользователи

Что Как
email + password hash bcrypt, 12 rounds
Сессия in-memory (TTL 30 дней). Restart сервера — все вылогинились
Сохранение образцов (bid_req, bid_res) AES-GCM ciphertext. Сервер видит байты, но не может расшифровать.
KDF salt + wrapped DEK + IVs Да. Бесполезно для атакера без твоего пароля.
Notes / metadata образца То же шифрование
Партнёры (label) То же шифрование
email_verified_at timestamp (только после verify-email)

Реализация: пароль на клиенте → PBKDF2 600k SHA-256 → KEK → разворачивает сохранённый DEK → DEK шифрует контент. Сервер не видит пароля и не может получить DEK. При смене пароля DEK перепаковывается (не перегенерируется), чтобы не потерять доступ к старым образцам.

Как читать результат

Каждая находка (finding) выглядит так:

{
  "id": "imp.banner.size_required",
  "level": "error",
  "path": "imp[0].banner",
  "params": { "num": 1 },
  "msg": "imp[1] banner: нужно указать w+h или format[]",
  "specRef": "https://iabtechlab.com/...#322"
}
  • level — error (нарушение обязательного правила) / warning (подозрительно, не запрещено) / info (заметка)
  • path — JSON-path до проблемного поля в твоём payload
  • msg — человеческий текст, локализованный
  • specRef — прямая ссылка на параграф IAB-спецификации

Общий статус:

  • errors — есть хотя бы один level: error
  • warnings — только warnings/info, но не clean
  • clean — ничего не найдено

Быстрый пример

Вставь в поле BID REQUEST:

{
  "id": "test-001",
  "imp": [
    { "id": "1", "banner": { "w": 300, "h": 250 }, "bidfloor": 0.5 }
  ],
  "site": { "id": "publisher-abc", "domain": "example.com" },
  "device": { "ua": "Mozilla/5.0", "ip": "1.2.3.4" }
}

Нажми analyze stream (или Ctrl+Enter) — увидишь detected version 2.5, несколько warnings (нет at, tmax, cur), spec-refs к IAB §3.2.1, §3.2.4.

Добавь в BID RESPONSE:

{
  "id": "test-001",
  "seatbid": [
    { "bid": [
      { "id": "b1", "impid": "1", "price": 1.5,
        "adm": "<html>creative</html>",
        "adomain": ["example.com"], "w": 300, "h": 250 }
    ]}
  ],
  "cur": "USD"
}

Теперь crosscheck активен — увидишь impid → imp.id match, match размеров, проверку price > bidfloor и ad preview в нижней правой карточке (sandboxed).
